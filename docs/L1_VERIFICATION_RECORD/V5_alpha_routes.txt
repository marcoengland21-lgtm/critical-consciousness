CAVEAT — this walk was accidentally passing
────────────────────────────────────────────────────────────────────────
At the time of this walk, the production application code was still the
pre-L1 build (commit 32e89f3, deployed yesterday). That code uses the
hardcoded DEFAULT_GROUP_ID = '00000000-...001' (test group) for every
group-scoped query. Alpha happens to be a single-group member of the
test group, so the hardcoded constant aligned with alpha's actual group
and every page rendered correct-looking data — but this walk did NOT
exercise the L1 resolver path, the membership lookup, or the per-user
group filtering. It was an accidental pass against the wrong code.

The bug was caught when V5 beta started: beta saw "Christchurch Capital
Reading Group" as the group name (returned by the hardcoded query
against a publicly-readable groups row) but with Watermelon-scoped data
(every group-scoped read was RLS-rejected because beta isn't a test-
group member). The mismatch between the name label and the data is the
fingerprint of pre-L1 code reading post-L1 schema.

This walk needs to be re-run AFTER the L1 application code is deployed.
The post-deploy alpha walk is the actual first real V5 alpha test.
The captures below are kept as a record of what we initially observed,
but they don't constitute V5 alpha verification on their own.

The structural pattern: every page's SystemStatusStrip should show the
test group ("Christchurch Capital Reading Group") since alpha is a
single-group member of test group only. Application queries should be
scoped to that group through the L1 RLS policies.
